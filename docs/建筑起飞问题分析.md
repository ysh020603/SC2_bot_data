# 建筑「起飞后不降落」问题分析

> 适用策略：`battle_cruisers`（LLM macro pipeline）
> 复现录像：`game_records/battle_cruisers_test/20260616_143916_bc_addonpos2/`
> 本文只做**问题定位**，不含修复改动。

---

## 1. 现象

游戏录像中可见多座生产建筑（Factory / Starport）**起飞后悬停在空中、不再降落**，
直到被打掉。日志中体现为 `FACTORYFLYING` / `STARPORTFLYING` 一旦出现就长期存在：

```
06:05  Completed: ... 1 FACTORY, 1 FACTORYFLYING, ...
06:53  Completed: ... 1 FACTORY, 1 FACTORYFLYING, ... 1 STARPORT, 1 STARPORTFLYING, ...
09:13  Completed: ... 1 STARPORTFLYING ...     # 一直没落回来
终局   [GameAnalyzerEnd] FACTORYFLYING total: 1   （整局飞行状态）
       [GameAnalyzerEnd] STARPORTFLYING total: 1
```

同时 `BuildAddon` 反复对**不同坐标**下达挂件指令，说明建筑在空中四处找落点：

```
06:05  [BuildAddon] UnitTypeId.STARPORTTECHLAB to (29.0, 130.0)
06:07  [BuildAddon] UnitTypeId.STARPORTTECHLAB to (44.0, 134.0)
06:54  [BuildAddon] UnitTypeId.BARRACKSTECHLAB to (42.0, 142.0)
06:54  [BuildAddon] UnitTypeId.BARRACKSTECHLAB to (35.0, 135.0)
```

用户描述：**「起飞 → 返回选址 → 不下落」**，与日志一致。

---

## 2. 根因链

整条链由「建挂件」触发，核心是 **SC2 引擎在建筑右侧挂件位被占用时会自动 LIFT 建筑去找空位**，
而我们没有任何代码能可靠地把它降回来。

### 2.1 触发点：下达挂件建造指令

调度器把 `BUILD_TECHLAB_*` / `BUILD_REACTOR_*` 这类挂件动作路由到 sharpy 的 `BuildAddon`：

- `SC2_Agent/execution/scheduler.py:330-331` — `CAT_ADDON` 走 `_issue_build_or_research`
- `SC2_Agent/execution/scheduler.py:396-397` — 创建 `BuildAddon` act
- `SC2_Agent/execution/mapping.py` 中 `make_addon_act()` — 把动作名映射成 `BuildAddon(addon_type, from_type, to_count)`

`BuildAddon.execute()` 的关键逻辑：

```59:65:sharpy/plans/acts/terran/build_addon.py
                if await self.ai.find_placement(UnitTypeId.SUPPLYDEPOT, center, 0, False):
                    self.tried_to_build_dict[builder.tag] = self.ai.time
                    self.print(f"{self.unit_type} to {center}")
                    builder.build(self.unit_type)
                else:
                    self.print("no space")
```

问题有两处：

1. **预检不精确**：`center = builder.position.offset((2.5, -0.5))`，用 **2x2 SupplyDepot footprint** 去近似挂件占地。
   SupplyDepot 能放下不代表挂件能原地附着——预检通过但实际附着位被占。
2. **`builder.build(self.unit_type)` 不带坐标**：当建筑自己右侧的挂件位被相邻建筑占住时，
   SC2 引擎**自动让该建筑起飞（LIFT）去别处找挂件空位**。这就是 `FACTORYFLYING` / `STARPORTFLYING` 的来源。

### 2.2 深层原因：选址没有为挂件预留右侧空位

生产建筑（Barracks/Factory/Starport）由 `GridBuilding` 选址：

- `sharpy/plans/acts/grid_building.py:337-377` — `position_terran()` 遍历 `building_solver.buildings3x3`
- `sharpy/plans/acts/grid_building.py:308-335` — `_validate_terran_point()` 只校验「建筑自身 3x3 能不能放」

`_validate_terran_point` **没有检查该建筑右侧 2 格的挂件位是否会被后续/相邻建筑占用**。
于是多座生产建筑可能紧贴排布，彼此挤掉对方的挂件位。等到 2.1 要建挂件时，右侧无空位 → 引擎 LIFT。

> 注：本轮曾尝试在 `_validate_terran_point` 末尾用 `can_place_single(addon_type, ...)` 预留挂件位，
> 但 SC2 的 placement query 对挂件 unit type 几乎恒返回 False，导致**连 Barracks 都选不出落点**
> （日志刷屏 `Can't find free position to build BARRACKS in!`，资源堆到 4600+），已回退。
> 该尝试见 `game_records/.../20260616_143622_bc_addonpos/`。

### 2.3 兜底失效：起飞后没有可靠的降落路径

为召回起飞建筑新增了 `LandFlyingProduction`（挂在 `SKILL/terran/background_tactics.py:66`）：

```66:79:sharpy/plans/tactics/terran/land_flying_production.py
    def _closest_free_slot(self, near: Point2, occupied: Set[Point2]) -> Optional[Point2]:
        best: Optional[Point2] = None
        best_dist = float("inf")
        for point in self.building_solver.buildings3x3:
            if point in occupied:
                continue
            # avoid landing on top of an existing structure or addon footprint
            if self.ai.structures.closer_than(1, point):
                continue
            dist = near.distance_to(point)
            if dist < best_dist:
                best_dist = dist
                best = point
        return best
```

它失效的原因：

1. **只在 `buildings3x3` 已知格里找空位**，而中后期这些格子基本被现有建筑占满 → `_closest_free_slot` 返回 `None` → 不下令降落。
2. **它把降落点选在和挂件区无关的任意空格**，即使落下也丢失了原本要挂的挂件位，与 `BuildAddon` 的意图冲突，会再次起飞。
3. 与 `BuildAddon` 不协调：`BuildAddon` 还在持续对飞行建筑发 `builder.build(addon)`（2.1），
   等于「一边想降落、一边又被命令起飞找挂件位」，互相打架，建筑卡在空中。

## 4. 可能的修复方向（仅记录，未实现）

1. **选址阶段预留挂件位（首选）**
   在 `_validate_terran_point` 中用「右侧 2 格的几何占用判断」而非 `can_place_single(addon)`，
   即：候选 3x3 点的右侧 `+ (2.5, -0.5)` 处，要求不与其它生产建筑/其挂件区重叠，
   且用 `find_placement(SUPPLYDEPOT, right, 0, False)` 验证可放。避免 2.2 里 `can_place_single` 恒 False 的坑。

2. **挂件指令带显式坐标 + 原地校验**
   `BuildAddon` 改为：只有当建筑**当前右侧**挂件位可放时才 `builder.build(addon, position=...)`，
   否则不发指令（宁可等待，也不让引擎自动起飞）。

3. **修好降落兜底**
   `LandFlyingProduction` 改为：让飞行建筑**落回它自己原来的位置**（或带挂件位的合法点），
   且与 `BuildAddon` 互斥（飞行中的建筑本帧不再被 `BuildAddon` 下挂件指令），打破「起飞↔降落」拉锯。

4. **复用 sharpy 既有 AddonSwap**
   `addon_swap.py` 已实现完整的 lift→move→land 状态机（`ExecuteAddonSwap`），
   但当前 LLM 管线的 background tactics 未启用它。可评估接入以替代自写的 `LandFlyingProduction`。

## 5. 2026-06-16 修复记录

本轮已按“建筑建造时预留挂件空间，挂件仍用 unit 形式下单、不指定位置”的方向修改。

### 已做改动

1. **生产建筑选址预留挂件位**
   - `sharpy/plans/acts/grid_building.py`
   - Barracks / Factory / Starport 选址时检查右侧 add-on 位置。
   - 使用 `find_placement(SUPPLYDEPOT, addon_center, 0, False)` 作为挂件 2x2 可放置近似校验。
   - 增加几何碰撞检查，避免新建筑占用已有生产建筑的挂件位。
   - 将挂件安全距离从过大的 `6.0` 调整为 `3.0`，避免基地建筑铺得过散，同时保留右侧挂件空间。

2. **BuildAddon 不再触发飞行建筑**
   - `sharpy/plans/acts/terran/build_addon.py`
   - 只选择 ready / idle / landed / no add-on 的生产建筑。
   - 挂件仍执行 `builder.build(addon_unit_type)`，不指定位置。
   - 下单前检查当前右侧挂件位可用；不可用则等待，不向 SC2 引擎发会触发起飞的挂件指令。
   - 增加短时间 pending 计数，避免多个建筑同时抢同一个挂件目标。

3. **删除科技链自动补前置 action**
   - `SC2_Agent/execution/scheduler.py`
   - 科技不满足时只等待，不再调用 gap-fill 自动插入 Factory / Starport 等前置建筑。
   - 修复了“已有 Factory 在路上，scheduler 又为 Starport 插入第二个 Factory”的重复建筑来源。

4. **禁用起飞后自动找落点兜底**
   - `SKILL/terran/background_tactics.py`
   - 不再注册 `LandFlyingProduction`，避免建筑起飞后反复搜索落点并与 BuildAddon 互相打架。

5. **Supply Depot 智能插入**
   - `SC2_Agent/data_tools/supply_planner.py`
   - Stage5 仍保留原 supply threshold。
   - 新逻辑按后续连续 Train action 的人口需求计算补给，目标为“训练需求 + 4 supply 冗余”，避免逐个单位机械插 Depot。
   - Command Center / Orbital Command 的供给不再提前当作即时可用，避免扩张未完成时误判不缺补给。

6. **battle_cruisers 策略调整**
   - `SKILL/terran/battle_cruisers/Top_agent_0.md`
   - 提前二基地。
   - 增加 Engineering Bay、Missile Turret、更多 Marine 的中期防空。
   - 延后高额 Battlecruiser 科技，先确保防守和经济。

### 当前结论

本轮关注的旧问题已经未复现：

- 生产建筑为建 add-on 起飞。
- 起飞后反复搜索落点。
- scheduler 自动补前置导致重复 Factory。
- add-on 指令与降落兜底互相打架。

后续如果继续优化，应关注新的低优先级问题：同一帧多个 `GridBuilding` act 偶尔会打印相同候选坐标；目前未导致旧的起飞问题复现，但可进一步把“本帧已选建筑点”加入临时 reservation。
