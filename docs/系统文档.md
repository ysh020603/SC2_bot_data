## 0. 一句话概述

底层是基于规则的《星际争霸 II》Bot 平台 **Sharpy**（封装 `python-sc2`/`burnysc2`）。
在其之上，本项目用 **五个 LLM 智能体 + 一套 DATA_TOOLS 知识库 + 一个命令式执行调度器**，
把"自然语言战略"逐级降维成"按顺序执行、带资源/科技/补给约束的绝对增量动作序列"，
再交给 Sharpy 的执行原语落地。**当前仅适配人族（Terran）**。

```
玩家指令 / 策略库
   │
   ▼  Top Agent（t=0 选策略，仅一次）
strategy_description（Top_agent_0.md 全文）
   │
   ▼  每 60s 或「动作序列执行完 / 仅剩单等待者」触发一次宏观流水线 ↓↓↓
┌──────────────────────────────────────────────────────────────┐
│ Strategy Step Source    Top_agent_0.md 当前 [Step N] 文本       │
│ Stage2 Naming Agent     当前 obs + step → 标准 unit/upgrade 名 + 数量 │
│ Stage3 DATA_TOOLS 映射  标准名 → 标准 Action key                │
│ Stage4 Ordering Agent   当前 step + 前置/冲突/成本提示下排序     │
│ Stage5 Supply Planner   按阈值插入 build_supply_depot           │
└──────────────────────────────────────────────────────────────┘
   │  (action_name, quantity) 序列
   ▼
ExecutionScheduler（命令式调度，每帧 execute）
   ├─ build / research / addon → Sharpy Act（GridBuilding / BuildGas / Expand / Tech / BuildAddon）
   ├─ train / addon / morph     → Executor Agent（LLM）选执行单位
   ├─ 单槽等待队列 + 矿/气/人口预留 + 超车（overtake）
   ├─ 单等待者预取下一步（append）/ 序列耗尽续作（replace）
   ├─ 等待超时放弃（60s）/ RUNNING 卡死放弃（25s）
   ├─ 科技链路等待 / 自动补前置
   ├─ 建造防重复放置闸 + SCV 提前移动
        ║ 并行
        ▼
BackgroundTactics（后台战术，只保留**不消耗矿/气/supply**的工具：侦察/防守/集结/进攻/收尾/Mule/分矿等）
```

## 1. 与旧版（Top/Mid/Down 声明式）的区别

| | 旧版（legacy） | 新版（当前） |
|---|---|---|
| 规划粒度 | Mid 给"目标数量 `to_count`"（声明式） | `Top_agent_0.md` step 给固定阶段要求，Stage2 结合 obs 产生命令式增量 |
| 翻译链路 | Down Agent → action JSON | Naming Agent + DATA_TOOLS 映射 |
| 排序 | 无显式排序 | Ordering Agent（带当前 step、前置/冲突/成本提示） |
| 补给 | 战术 `AutoDepot` | Supply Planner 按阈值插入 |
| 执行 | `ActLLMOngoingTasks`（Sharpy Act 自判每帧续产） | `ExecutionScheduler`（单槽等待队列 / 资源保留+超车 / 科技等待 / RUNNING 超时 / SCV 预移动） |
| 选执行单位 | 由 Sharpy Act 内部决定 | train/addon/morph 由 Executor Agent 选；build/research 仍交 Sharpy |

旧的 `mid_agent.py` / `down_agent.py` / `ActLLMOngoingTasks` 仍保留在仓库中（向后兼容），
但**默认运行路径已切换到新流水线**。`--mid-model` / `--down-model` 标记为 legacy。

## 3. 五阶段宏观流水线（`_run_macro_pipeline_blocking`）

入口在 `dummies/generic/universal_llm_bot.py`。每个触发周期串行执行下列阶段，
全过程作为**一条交互记录**写入轨迹 JSON（含每阶段原始/解析结果）。

### 触发机制（`pre_step_execute`）
1. 距上次触发 ≥ `MACRO_POLL_INTERVAL`（默认 **60s**）→ 固定周期触发；
2. 或 当前动作序列已执行完（`scheduler.is_drained()`）且距上次触发 ≥ `MACRO_MIN_RETRIGGER`（默认 **5s**）→ **`replace`** 安装下一步；
3. 或 仅剩一个 `WAITING` 动作（`scheduler.is_single_waiter_remaining()`）且距上次触发 ≥ `MACRO_MIN_RETRIGGER` → **`append`** 预取下一步（保留等待者；**最后一步 `is_last` 不预取**；同一等待者只预取一次）。
- 每次触发都会先把**当前 obs 完整打印进 `.log`**（`[Observation @ decision]`），便于回看决策依据。

### Strategy Step Source（`Top_agent_0.md`）
- **输入**：当前策略文件 `# Details` 中的 `[Step N]` 文本。
- **输出**：当前 step 原文，直接传给 Stage2；不再调用额外规划 LLM，也不再生成 `append/replace` 计划。
- **推进规则**：初始传 Step 1；当当前 scheduler 中所有 action 执行完后传下一个 step（`replace`）；仅剩单 `WAITING` 时预取下一步（`append`）；最后一个 step 执行完后重复传最后一个 step。

### Stage 2 — Naming Agent（`naming_agent.py`）
- **输入**：当前 obs、当前 strategy step、人族标准 unit/upgrade 名集合（`terran_names.py`）、别名表。
- **输出**：`{"items": [{"name": "Barracks", "count": 1}, {"name": "Marine", "count": 2}]}`
- **要点**：把口语/别名规范成数据库标准名（`resolve_alias` + `is_known_terran_entity` 校验），
  无法识别的实体直接丢弃；supply depot 在此被过滤。

### Stage 3 — DATA_TOOLS 映射（`entity_to_actions.py`）
- 标准实体名 → 标准 **Action key**（如 `Barracks → TERRANBUILD_BARRACKS`、
  `Marine → BARRACKSTRAIN_MARINE`、`OrbitalCommand → UPGRADETOORBITAL_ORBITALCOMMAND`）。
- 纯工具调用，无 LLM。

### Stage 4 — Ordering Agent（`ordering_agent.py`）
排序 Agent 会同时读取当前 strategy step、当前 obs，以及 DATA_TOOLS 生成的三类提示：
- **Strategy Step**：来自 `Top_agent_0.md` 的当前 `[Step N]` 原文，用于理解战略优先级和预期时机；动作列表仍是权威来源，排序阶段不得因 step 增删 action；
- **前置提示**（`check_action_prereqs.tech_chain_relations`）：序列内动作之间的科技先后关系；
- **冲突提示**（`detect_action_conflicts`）：可能争抢同一执行单位的动作；
- **成本提示**（`action_cost`）：每个动作的矿/气/时间（帧→秒按游戏帧率换算）。
- **输出**：`{"ordered_actions": [...]}`。排序**忽略 supply depot**；解析失败时回退到输入顺序，
  并把被漏掉的动作补回。

### Stage 5 — Supply Planner（`supply_planner.plan_supply`）
- 沿排序后的序列**模拟 supply 走势**，当预测剩余补给跌破阈值（`SUPPLY_THRESHOLD`，默认 **8**）
  时插入 `TERRANBUILD_SUPPLYDEPOT`；若补给冗余则尽量少插，避免浪费。

最终得到 `[(action_name, quantity), ...]`，通过 `scheduler.set_actions(..., mode=append/replace)` 注入调度器。

## 4. 命令式执行调度（`execution/scheduler.py`）

`ExecutionScheduler(ActBase)` 每帧 `execute()`，维护一组 `PlannedAction`（`command.py`）。

### 4.1 PlannedAction 状态机

```
PENDING ──► RUNNING ──► DONE
   │            │
   │            └──► ABANDONED（RUNNING 超时，build/research/addon 卡死）
   │
   └──► WAITING ──►（前置/矿气/人口到位）──► 尝试下发
             │
             └──► ABANDONED（等待超时）
```

| 状态 | 含义 |
|---|---|
| `PENDING` | 尚未轮到或等待槽被占用，暂挂起 |
| `WAITING` | 阻塞于科技链、执行单位、矿/气或人口；`note` 区分具体原因 |
| `RUNNING` | 已调用 Sharpy Act 或已下达 train/addon/morph，尚未完成 |
| `DONE` | 本动作数量已满足 |
| `ABANDONED` | 超时放弃或无法映射/启动 act |

`WAITING` 的 `note` 示例：`waiting: resources`、`waiting: supply`、`waiting: tech missing`、`waiting: prerequisite in progress`、`waiting: no free producer` 等。

### 4.2 单槽等待队列（single-slot wait queue）

**同一时刻最多只有一个动作处于 `WAITING`。**

- 队列中**第一个**（按 `set_actions` 注入顺序）遇到阻塞的动作占据等待槽。
- 后续动作若也想进入等待，会被**降级回 `PENDING`**，note 为 `pending: wait slot occupied`。
- 每帧开头 `_enforce_single_waiter()` 清理重复等待者；`_get_waiter()` 取当前唯一等待者。

设计意图：避免多个昂贵动作同时「占坑等资源」，导致整段 macro 长时间无法 `is_drained()`、流水线无法推进到下一步。

**单等待者预取**：当 `is_single_waiter_remaining()` 为真时，宏观流水线可用 `append` 把下一步动作追加到队列末尾，而不必等等待者完成；Bot 用 `_prefetched_while_waiter_key` 防止对同一等待者重复预取。

### 4.3 资源保留 + 超车（overtake）

凡处于 **`WAITING`** 的等待者，统一对其成本做**矿 + 气 + 人口**预留：

```python
reserved_min, reserved_gas, reserved_supply = _waiter_reservation(waiter)  # 任意 WAITING
avail_min = ai.minerals - reserved_min - spent_min
avail_gas = ai.vespene - reserved_gas - spent_gas
avail_supply = current_free_supply(ai) - reserved_supply - spent_supply
```

- **`_live_cost`**：优先 `ai.calculate_cost(ability)` 取**实时增量成本**（正确处理 morph/addon 差价，例如升轨道只算 +150 矿）。
- **`_live_supply_cost`**：同上取 supply；回退 `pa.cost_supply`。
- **`current_free_supply`**（`supply_planner.py`）：当前空闲人口 + 已在途 supply depot 的 headroom。
- **`spent_min` / `spent_gas` / `spent_supply`**：本帧已成功下发的动作累计扣减，防止同帧重复「花同一笔钱/人口」。
- **超车**：等待者占住预留后，扫描队列中**排在它后面**的动作；若某动作所需矿/气/人口 ≤ 当前可用余额（已扣除预留与本帧已花费），则**先执行该动作**，不必等等待者攒够资源。

等科技的动作进入 `WAITING` 时**同样锁定矿/气/人口**，避免后续动作在账本里「透支」等待者即将使用的资源。

### 4.4 等待超时放弃

| 参数 | 调度器默认 | Bot 实际注入 | 作用 |
|---|---|---|---|
| `wait_abandon_sec` | `20.0` | **`60.0`**（`UniversalLLMBot.WAIT_ABANDON_SEC`） | `WAITING` 持续超过该秒数 → `ABANDONED`，释放等待槽与资源预留 |
| `running_abandon_sec` | `25.0` | 未覆盖，用默认 | `RUNNING` 的 build/research/addon 超过该秒数仍未完成（典型：找不到落点、act 每帧返回 False）→ `ABANDONED`，避免永久阻塞 macro；下轮流水线可重新请求 |

设为 `0` 表示永不因超时放弃。

### 4.5 科技链路等待 / 自动补前置（`prereq_runtime.py`）

每帧对每个非终态动作先做 `is_available_now`：

1. **前置已在建造/研究中**（`chain_in_progress`）→ 占等待槽，`WAITING`（note: prerequisite in progress），满足后立即下发。
2. **前置缺失且未在进行** → `_insert_gap_fills` 在队列当前位置**前插**缺失前置动作（去重）；插入后 `break` 本帧循环，下帧再扫。
3. 无法插入且仍不满足 → `WAITING`（note: tech missing）。

### 4.6 SCV 提前移动（pre-move）

`WAITING` 的 **build** 动作在矿/气/人口不足时，调用 `_premove_scv`：用 `find_placement` 选点并把 SCV `move` 到建造点附近，资源到位后减少首帧卡顿。

### 4.7 建造防重复放置闸

`GridBuilding` 依赖 `already_pending` 判断「是否已下单」，对「SCV 刚接到指令尚在赶路」有数帧延迟，可能导致同一「建 1 座」请求连续下单。

调度器在 `_issue_build_or_research` 中用 `_existing_plus_en_route()` **直接扫描 worker 携带的建造指令**（无延迟），若已有+在途数量 ≥ 目标数，立即标 `DONE`，不再让 act 重复下单。

### 4.8 执行分派

- **build / research / addon**：惰性实例化 Sharpy Act，**不经过 Executor LLM**：
  - `Refinery → BuildGas`；`CommandCenter → Expand`；其它建筑 → `GridBuilding`；升级 → `Tech`；挂件 → `BuildAddon`。
  - `to_count` = 现存数量 + 本动作增量。
- **train / morph**（及未走路由的 addon）：走 **Executor Agent**：
  1. `executor_select.candidate_executors` 规则筛选能执行该 ability 的单位；
  2. 生成冲突提示（候选是否也被后续动作需要）；
  3. `executor_agent.py` 让 LLM 在候选里选一个，带 3s 缓存与 idle 回退。

> Executor 的提示词显式要求：**不要一味追求效率而阻碍后续冲突动作**
> （例如没挂 add-on 的 Barracks 先去造 Marine，会导致短期无法挂 Tech Lab 而卡住）。

### 4.9 每帧主循环（简图）

```
enforce_single_waiter()
waiter = get_waiter()
abandon_waiter_if_timed_out(waiter)      # WAITING 超 60s
abandon_stuck_running()                  # RUNNING 超 25s

for pa in actions (按序):
    if pa 终态: continue

  [1] 科技门 ──不满足──► claim_wait_slot(WAITING) 或 insert gap_fill
  [2] 资源+人口门 ──不足──► premove_scv(build); claim_wait_slot(WAITING)
              ──足够──► try_issue → spent += cost (min/gas/supply)
              （后续动作可用 avail - reserved - spent 超车）
```

## 5. 后台战术 + 按 skill 注入的侦察/攻击

`create_plan()` 返回三条**并行**轨：`ExecutionScheduler` + 后台战术 + 侦察/攻击战术。
全部**只保留不消耗矿/气/supply 的工具**（不与调度器抢资源）。

### 5.1 通用后台战术（`SKILL/terran/background_tactics.py`）
策略无关、始终运行，只含**运营 + 防守**：
`MineOpenBlockedBase`、`PlanCancelBuilding`、`LowerDepots`、`DistributeWorkers`、
`SpeedMining`、`ContinueBuilding`、`CallMule`、`PlanZoneDefense`、
`PlanWorkerOnlyDefense`、`ManTheBunkers`。

### 5.2 按 skill 注入的侦察 / 攻击
侦察与进攻是**策略相关**的，已从后台战术中抽出，改为**每个 skill 用自己的 python 文件注入**：

| 来源 | 说明 |
|---|---|
| `SKILL/terran/<strategy>/scout_attack.py` | **优先**：该 skill 自定义的侦察/攻击触发 |
| `SKILL/terran/scout_attack_default.py` | **兜底**：通用 macro 风格（`WorkerScout` / `ScanEnemy` / `PlanZoneGatherTerran` / `PlanZoneAttack(60)` / `PlanFinishEnemy`） |

加载逻辑见 `UniversalLLMBot._load_scout_attack_tactics`：按 `选中策略 → 默认` 顺序查找，
取模块里第一个 `BuildOrder` / `SequentialList` 子类实例化。

**示例（marine_rush）**：`SKILL/terran/marine_rush/scout_attack.py` 参考手写 dummy
`dummies/terran/marine_rush.py`，定义了 `DodgeRampAttack`（躲 FORCEFIELD 的
`PlanZoneAttack` + 小撤退）并以**更低的进攻阈值**（默认 10）实现 rush 激进打法。

新增一个策略的侦察/攻击：在该策略目录放一个 `scout_attack.py`，定义一个
`BuildOrder` 子类（构造函数可选 `attack_value`）即可，无需改 bot 代码。

**显式排除**会消耗资源的 `AutoDepot` / `MorphOrbitals` / `Repair` / `DefensiveBuilding`
——这些职责已由流水线 / 调度器接管。各工具的资源消耗分类详见
[`SKILL/terran/scout_and_attack_tools.md`](../SKILL/terran/scout_and_attack_tools.md)。

## 6. 关键可调参数

`dummies/generic/universal_llm_bot.py`：

| 常量 | 默认 | 含义 |
|---|---|---|
| `MACRO_POLL_INTERVAL` | `60.0` | 宏观流水线固定触发周期（秒） |
| `MACRO_MIN_RETRIGGER` | `5.0` | 序列耗尽或单等待者预取的最短再触发间隔（秒） |
| `WAIT_ABANDON_SEC` | **`60.0`** | `WAITING` 放弃超时（秒）；`0` = 永不放弃 |
| `running_abandon_sec` | `25.0`（仅调度器构造参数，Bot 未覆盖） | `RUNNING` 的 build/research/addon 卡死放弃超时（秒） |
| `SUPPLY_THRESHOLD` | `8.0` | 预测剩余补给低于此值则插 supply depot |

`bot_loader/game_starter.py`：`game_time_limit` 默认 30 游戏分钟，可用环境变量
`SC2_GAME_TIME_LIMIT=<秒>` 覆盖（冒烟测试 / 控制时长很有用）。

## 7. LLM 配置

模型池在 `API_config/config.json` 的 `llm_agents_pool`。每个 key 含
`api_url / api_key / model_name / temperature / is_reasoning(thinking 开关)`。
`API_Tools/llm_caller.py` 按 `is_reasoning` 向厂商 API 注入 thinking 开关并自动剥离 `<think>` 段。

**当前默认**：五个 LLM 阶段（含 legacy 三层）全部使用 **`DeepSeek-V4-flash`（`is_reasoning: false`，不开 thinking）**。
对应 `run_vs_ai.py` 顶部：

```python
DEFAULT_TOP_MODEL = DEFAULT_NAMING_MODEL \
  = DEFAULT_ORDERING_MODEL = DEFAULT_EXECUTOR_MODEL = "DeepSeek-V4-flash"
```

可分别用 `--naming-model / --ordering-model / --executor-model / --top-model`
覆盖单个阶段（取 `config.json` 里的 key）。

## 8. 运行

> 环境搭建详见 [`docs/环境配置教程.md`](环境配置教程.md)。以下假设已在 `SC2_0615` 环境内、
> 且已 `export SC2PATH=/path/to/StarCraftII/`。

### 单局
```bash
conda activate SC2_0615
export SC2PATH=/data2/SC2/StarCraftII/
python run_vs_ai.py --enemy-difficulty easy --batch-name demo
# 短时冒烟（限制游戏时长，便于快速验证轨迹保存）
SC2_GAME_TIME_LIMIT=240 python run_vs_ai.py --enemy-difficulty easy --batch-name smoke
```

常用参数：`--bot-race/--enemy-race`、`--enemy-difficulty/--enemy-build`、
`--force-strategy <name|none>`、`--real-time`、`--batch-name/--run-index`。

### 批量
```bash
bash start_experiments.sh                 # 预设参数 + tmux 批量
bash run_vs_ai_batch.sh <总局数> <并发数> [fg|tmux]
python parse_sc2_logs.py                  # 批次胜率统计
```

## 9. 对局产出与轨迹 JSON

每局结束（`LLMObservationRecorder.on_end`）在
`game_records/<batch_name>/<match_id>/` 写出：

| 文件 | 内容 |
|---|---|
| `<match_id>.log` | 完整运行日志，含 `[UniversalLLMBot][LLM-INFER]` 各阶段行（每次 strategy step 下发前含完整 obs） |
| `<match_id>.SC2Replay` | 录像 |
| `<match_id>.json` | **LLM 交互完整记录（关键产物）** |
| `<match_id>.llm_calls.json` | **每一次 LLM 调用的原始 prompt 与 output**（按调用顺序记录，含 agent/model_key/game_time，边运行边落盘） |

### `*.llm_calls.json` 结构
```json
{
  "match": "<match_id>",
  "llm_call_count": 15,
  "calls": [
    { "seq": 1, "game_time": 0.0, "macro_cycle": 1,
      "agent": "naming", "model_key": "DeepSeek-V4-flash",
      "prompt": [{"role": "system", "content": "..."}, {"role": "user", "content": "..."}],
      "output": "..." }
  ]
}
```

### 轨迹 JSON 结构
```json
{
  "metadata": { "result": "Victory|Defeat|Tie", "matchup": "...",
                "game_duration_seconds": 240.0, "interval_seconds": 60.0,
                "llm_interaction_count": 11, "record_count": 8 },
  "interactions": [
    { "trigger_reason": "top_agent_initial_t0_forced", "top_agent_initial": {...} },
    { "cycle": 1, "trigger_reason": "initial_step", "mode": "replace",
      "strategy_step": {"number": 1, "text": "..."},             // Step source
      "strategy_step_text": "...",
      "naming_raw": "...", "named_items": [...],               // Stage2
      "mapped_actions": {...},                                 // Stage3
      "ordering_raw": "...", "ordered_actions": [...],         // Stage4
      "ordered_with_supply": [...],                            // Stage5
      "installed_pairs": [["TERRANBUILD_BARRACKS", 1], ...],   // 注入调度器
      "observation_at_this_moment": "...",                     // 决策时英文 obs
      "observation_structured": {...},                         // 结构化快照
      "top_agent_strategy": "battle_cruisers", "wall_elapsed_seconds": 4.37 }
  ]
}
```

> 五阶段原始与解析结果齐全。观测字段完整说明见
