# SC2 Agent 测试运行流程记录

本文记录单局实验的可复用流程。当前推荐用于模型评估的基线是：`name` / `order` 两个环节使用 `DeepSeek-V4-pro`，其余环节使用 `DeepSeek-V4-flash`，并关闭 supply 托管，让 LLM 自己决定补给站排序。

## 测试原则

- 本测试用于评估不同 LLM 的策略理解和排序能力，不对 LLM 的漏项、错项、空输出做代码兜底。
- 如果 LLM 输出本身不符合策略要求，记录为模型表现；不要通过 Agent 代码自动补动作来“修好”结果。
- 排查重点是 Agent 执行机制：当 LLM 已经输出合法动作后，检查 scheduler 是否正确保留、等待、下发、计数、deferred、DONE。
- `Stage4 ordering gaps` 中的 `missing` / `dropped` 是重要评估信号，不应被自动补齐。

## 推荐启动方式

Windows 下直接用很长的 `run_vs_ai.py` 默认 match id 容易触发日志路径过长。推荐用一个短 match id 的临时 launcher。模板如下，保存到 `tools\run_<test_name>.py`，跑完删除。

```python
"""Temporary launcher for one short-path test match."""

import os
import sys

sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))

import run_vs_ai
from dummies.generic.universal_llm_bot import UniversalLLMBot


UniversalLLMBot.SUPPLY_MANAGED = False


def short_match_id(**kw):
    return f"{kw['timestamp']}_pro_name_order_nosupply"


run_vs_ai.build_match_id = short_match_id

run_vs_ai.play_vs_ai(
    enemy_race="terran",
    enemy_difficulty="hard",
    enemy_build="air",
    top_model="DeepSeek-V4-flash",
    mid_model="DeepSeek-V4-flash",
    down_model="DeepSeek-V4-flash",
    naming_model="DeepSeek-V4-pro",
    ordering_model="DeepSeek-V4-pro",
    executor_model="DeepSeek-V4-flash",
    batch_name="model_eval_pro_name_order_nosupply",
    output_base_dir=r"C:\code\SC2_Agent_OLD\game_records",
    skip_version_update=True,
)
```

启动命令：

```powershell
$env:PYTHONUTF8='1'
$env:PYTHONIOENCODING='utf-8'

$py='C:\Users\Descfly\.conda\envs\SC2_0615\python.exe'
$out=Join-Path (Get-Location) 'game_records\model_eval_pro_name_order_nosupply_stdout.log'
$err=Join-Path (Get-Location) 'game_records\model_eval_pro_name_order_nosupply_stderr.log'

$p=Start-Process -FilePath $py `
  -ArgumentList @('tools\run_model_eval_pro_name_order_nosupply.py') `
  -WorkingDirectory (Get-Location) `
  -RedirectStandardOutput $out `
  -RedirectStandardError $err `
  -PassThru `
  -WindowStyle Hidden

"PID=$($p.Id)"
```

```powershell
Select-String -Path game_records\model_eval_pro_name_order_nosupply_stderr.log `
  -Pattern 'Strategy step|Stage2 named|Stage3 mapped|Stage4 ordering gaps|Stage5 SUPPLY_MANAGED|Scheduler active queue|DirectBuild|Abandoned stuck|Result for player|Result:' |

  ForEach-Object { $_.Line }
```

期望流程：

- Step 1 到 Step 6 依次出现。
- Step 7 出现后会重复出现。
- 推荐评估配置下应看到 `Stage5 SUPPLY_MANAGED=False: using LLM ordering as-is`。

- `Scheduler active queue after install` 是真实进入 scheduler 的队列，优先看它，不只看 requested pairs。
- 普通建筑多数量测试要看 `[DirectBuild] ... target=... progress=... existing=... en_route=... fresh=...`，确认每个 PA 独立推进。

## 常见坑

- `UnicodeEncodeError`：先设置 `$env:PYTHONUTF8='1'` 和 `$env:PYTHONIOENCODING='utf-8'`。
- 路径过长导致 `.log` 创建失败：使用短 `match_id`，并放在 `game_records\<batch_name>\<short_match_id>`。
- `conda` 不在 PATH：直接调用 `C:\Users\Descfly\.conda\envs\SC2_0615\python.exe`。
- `Start-Process -ArgumentList @('-c', '<多行代码>')` 容易被 Windows 参数拆坏：用临时 launcher 脚本更稳。
- `Action module SKILL.terran.Action not found; action space will be empty.` 本次测试中出现过，但不影响当前 step pipeline 跑完。
- `Stage4 ordering gaps` 出现 `missing`：先记录为 LLM 漏项，不要在代码里补齐；只有当 LLM 已经输出了动作但 scheduler 没执行，才进入执行机制排查。
- `Traceback` 若来自 stdout/stderr reader 线程且游戏继续运行，先归为日志解码问题；若影响对局进程，再排查启动器。
- 临时 launcher 跑完要删除，避免污染工具目录。

## 修改测试参数

常改项在 launcher 的 `play_vs_ai(...)` 中：

- `enemy_race`: `"terran" | "zerg" | "protoss" | "random"`
- `enemy_difficulty`: `"medium" | "hard" | "harder" | "veryhard"` 等
- `enemy_build`: `"air" | "macro" | "rush" | "timing" | "power"`
- `map_name`: 例如 `"KairosJunctionLE"`、`"AutomatonLE"`；必须是 `bot_loader/game_starter.py` 打印的已安装地图名
- `*_model`: 改成 `API_config/config.json` 里存在的 model key
- `batch_name`: 建议每组测试一个独立名字
- `short_match_id`: 保持短，包含核心变量即可

**模型评估推荐模型组合**：

- `naming_model="DeepSeek-V4-pro"`
- `ordering_model="DeepSeek-V4-pro"`
- `top_model / mid_model / down_model / executor_model = "DeepSeek-V4-flash"`

**Supply 托管实验参数**：`UniversalLLMBot.SUPPLY_MANAGED`（默认 `True`）控制 Stage5 行为。当前评估 LLM 能力时推荐在 launcher 顶部直接设置：

```python
from dummies.generic.universal_llm_bot import UniversalLLMBot

UniversalLLMBot.SUPPLY_MANAGED = False
```

- `True`（默认）：LLM 可以输出 depot，Stage5 提取后由算法按阈值重新插入，更稳定。
- `False`：直接使用 LLM 排好的序列（depot 放在 LLM 认为合适的位置），结果记录在 `supply_trace` 字段；用于评估模型时优先使用。

换地图回归时，只改 `map_name`，例如：


关键检查命令：

```powershell
Select-String -Path game_records\battle_cruisers_pro_name_order_nosupply_stderr.log `
  -Pattern 'Stage2 named|Stage3 mapped|Stage4 ordering gaps|Stage5 SUPPLY_MANAGED|Scheduler active queue|FACTORYFLYING|STARPORTFLYING|BARRACKSFLYING|BuildAddon|DirectBuild|Abandoned stuck|Result:' |

  ForEach-Object { $_.Line }
```

预期：

- Factory Tech Lab 出现 `BuildAddon UnitTypeId.FACTORYTECHLAB ...`。
- Starport Tech Lab 出现 `BuildAddon UnitTypeId.STARPORTTECHLAB ...`。
- `Stage4 ordering gaps` 为空；若不为空，记录为模型输出问题。
- `Stage5 SUPPLY_MANAGED=False`，确认没有 supply 托管。
- `Scheduler active queue after install` 中应保留 LLM 输出的关键动作。
- 不应出现 `FACTORYFLYING`、`STARPORTFLYING`、`BARRACKSFLYING`。

已验证：

- `KairosJunctionLE` / `Terran medium air`：FactoryTechLab、StarportTechLab、FusionCore 正常，未复现起飞。
- `AutomatonLE` / `Terran medium air`：FactoryTechLab、StarportTechLab 正常，未复现起飞。
